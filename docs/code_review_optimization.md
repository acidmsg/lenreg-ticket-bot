#### C. Дублирование шаблона сообщения о слотах

Оставшееся дублирование касается **ТОЛЬКО** сборки заголовка и шапки сообщения (врач, пациент, ссылка).
Также, из-за T6, указанные номера строк в [`src/services/monitor.py`](src/services/monitor.py) и [`src/handlers/common.py`](src/handlers/common.py) могут быть сдвинуты.

**Рекомендация:** Вынести в [`src/utils/helpers.py`](src/utils/helpers.py) функцию `format_notification_text(...)` для сборки заголовка и шапки.

#### D. `doctor_manager.py` — избыточная прослойка (24 строки)

**Где:** `src/database/doctor_manager.py` (весь файл)

`DoctorManager` — это тонкая обёртка, которая:

- Создаётся только для передачи в `discovery_loop`

**Экономия:** ~24 строки + 1 файл.

### 2.2 Умеренные изменения (Medium Risk, 50-200 строк)

#### E. Дублирование send_notification / send_nav_photo

**Где:**

- [`src/services/monitor.py`](src/services/monitor.py) — `_send_notification` (строки 18-53, 36 строк)
- [`src/handlers/common.py`](src/handlers/common.py) — `_send_nav_photo` (строки 156-241, 86 строк)

Обе функции:

1. Удаляют старое сообщение
2. Отправляют новое (send_photo или send_message)
3. Сохраняют message_id в БД

**Различия после T4/T5 (изображения-заголовки):**

| Параметр | `_send_notification` | `_send_nav_photo` |
|---|---|---|
| **Тип сообщения** | Текстовое + кнопка | Фото + текст + кнопка |
| **Ключ кэша** | `message_id` | `__nav__` (специфичный для навигации) |
| **Fallback** | Нет | При отсутствии бота удаляет старое и отправляет новое без фото |
| **Основное назначение** | Уведомления о слотах | Навигация по клиникам/врачам |

**Рекомендация:** Вместо полного слияния — выделить только общий низкоуровневый фрагмент, отвечающий за удаление старого сообщения, отправку нового (фото или текст) и сохранение `message_id` в БД. Этот фрагмент можно вынести в хелпер, а специфичную логику оставить в исходных функциях.
**Экономия:** ~20-30 строк.

#### F. Дублирование построения текста статуса в toggle_doctor

**Где:** [`src/handlers/common.py`](src/handlers/common.py), строки 670-731

`toggle_doctor` (при включении мониторинга) содержит:

1. Отправку "загрузочного" сообщения (строки 675-678)
2. `api.check_slots()` (строка 681)
3. Сохранение в кэш (строки 684-686)
4. Форматирование результата (строки 691-711)
5. Отправку финального сообщения с изображением (строки 719-731)

Эта логика частично дублирует логику `monitor_loop`.
**Примечание:** После сессии T6 `toggle_doctor` при включении вызывает `_send_nav_photo()` для обновления клавиатуры — этой логики нет в `monitor_loop`. Это усложняет вынесение общего хелпера. Также, из-за T6, указанные номера строк могут быть сдвинуты.

**Экономия:** ~25-30 строк.

#### G. `_start_background_tasks` — избыточность

**Где:** `src/main.py`, строки 182-220

Функция:

1. Запускает `monitor_loop` (1 задача)
2. Читает `clinic_ids` из БД
3. Для каждой клиники запускает `discovery_loop` (N задач)
4. Запускает `healthcheck_loop` и `cleanup_loop`

Можно упростить: вместо запуска per-клиника discovery_loop, запустить один общий discovery_loop, который сам итерирует clinic_ids. Это сэкономит ~20 строк в main.py и 10 строк в doctor_discovery.py.

**Экономия:** ~30 строк.

#### H. main.py — логика автоопределения прокси (76 строк)

**Где:** `src/main.py`, строки 42-118

`_generate_docker_gateways` + `_discover_proxy` + `_probe_host` — 76 строк для сканирования Docker gateway'ей. Функционал специфичный, редко используемый.

**Рекомендация:** Вынести в отдельный модуль `src/utils/proxy_discovery.py`.
**Экономия:** ~76 строк из main.py (не сокращение, а декомпозиция).

### 2.3 Архитектурные вопросы (High Risk, 200+ строк)

#### I. Database + DatabaseManager — можно ли объединить?

**Текущая архитектура:**

- [`src/database/database.py`](src/database/database.py) (658 строк) — низкоуровневый SQLite-движок
- [`src/database/manager.py`](src/database/manager.py) (229 строк) — кэширующая обёртка с `asyncio.Lock`
- [`src/database/doctor_manager.py`](src/database/doctor_manager.py) (24 строки) — тонкая прослойка (уже рекомендовано удалить)

**Проблема:**

- 12 методов в DatabaseManager — чистые прокси к Database (ниже)
- Кэш синхронизируется сложно: `update_user` делает read → modify → write + refresh
- Методы `data` (копия всего кэша) — потенциально дорогие

**Методы-прокси в DatabaseManager:**

| Метод DatabaseManager | Прокси к Database |
|----------------------|-------------------|
| `get_doctors_for_clinic` | `get_clinic_doctors` |
| `merge_doctors` | `merge_doctors` |
| `get_clinic_name` | `get_clinic_name` |
| `get_all_clinic_names` | `get_all_clinic_names` |

**Влияние Redis (2026-05-14):**
После внедрения [`src/utils/redis.py`](src/utils/redis.py) и [`src/utils/cache.py`](src/utils/cache.py), мониторинговый кэш (например, `swap_cache_key`, `is_spam`) был перенесён в Redis. Роль `DatabaseManager` теперь ограничена пользовательским кэшем (patients, monitoring settings, last_messages). Это делает объединение `Database` и `DatabaseManager` **МЕНЕЕ выгодным**.

**Рекомендация:** Оставить как есть. Разделение оправдано для тестов (можно мокировать DatabaseManager), а выгода от объединения снизилась.
**Экономия:** 0 строк (ранее ~100-200 строк, сейчас неактуально).

#### J. healthcheck.py + monitor.py — можно объединить?

**Текущая архитектура:**

- `monitor.py` (210 строк) — проверка слотов и уведомления

## 4.1. Неучтённые изменения, влияющие на анализ

Этот отчёт был изначально создан 2026-05-15, и с тех пор в проект были внесены значительные изменения, которые повлияли на актуальность некоторых находок:

1. **Внедрение Redis (2026-05-14):** Добавление [`src/utils/redis.py`](src/utils/redis.py) и [`src/utils/cache.py`](src/utils/cache.py) создало новый слой кэширования. Это фундаментально меняет контекст находок C, E, F, I, особенно снижая выгоду от объединения `Database` и `DatabaseManager`.
2. **Сессия T6 (format_slots, 2026-05-15):** Внедрение функции `format_slots()` в [`src/utils/helpers.py`](src/utils/helpers.py) частично решило проблему дублирования шаблона сообщения (Находка C) путем выноса логики форматирования списка слотов.
3. **Сессии T4/T5 (изображения-заголовки):** Добавление навигационной логики и изображений-заголовков в `_send_nav_photo` (Находка E) усложнило простое слияние с `_send_notification`.
4. **ErrorNotifier middleware:** Внедрение нового слоя `ErrorNotifier` (см. [`src/middleware/error_boundary.py`](src/middleware/error_boundary.py) и [`src/services/error_notifier.py`](src/services/error_notifier.py)) не было учтено в исходных метриках и может влиять на общую структуру проекта.

## 5. Вывод

Кодовая база в хорошем состоянии. Основные проблемы:

1. **Дублирование констант и мелких блоков** (nav_type_map, выбор города) — легко исправить
2. **DoctorManager как избыточная абстракция** — можно удалить без потерь
3. **Дублирование отправки сообщений** — можно объединить в утилиту
4. **DatabaseManager** — спорный, требует обсуждения

Проект не перегружен — соотношение кода к функционалу адекватное. Оптимизация на 5-10% возможна без риска.
