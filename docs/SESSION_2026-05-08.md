## Часть 1. Внедрение SQLite (ночная сессия)

**Результат:** 64/64 тестов проходят, 0 Pylance-ошибок

### 1. `database/database.py` — единый SQLite-движок

Создан новый модуль с классами:

**`Database`** — SQLite-обёртка с:

- `connect()` / `close()` — управление подключением (WAL-режим, busy_timeout)
- `_create_tables()` — автосоздание таблиц при первом подключении
- `migrate_from_json()` — миграция существующих JSON-данных в SQLite
- `get_user()` / `ensure_user()` — чтение/создание пользователей
- `update_user_field()` / `update_extra()` — обновление полей
- `get_clinic_doctors()` / `upsert_doctor()` / `upsert_clinic()` / `merge_doctors()` — работа с врачами
- `get_all_user_ids()` / `get_all_clinic_ids()` — вспомогательные запросы

**Схема таблиц:**

```sql
users:       uid PK, patients(JSON), monitoring(JSON), last_messages(JSON), last_notification_id, extra(JSON)
clinics:     clinic_id PK, name
doctors:     clinic_id, doctor_id, name, specialty  (PK: clinic_id+doctor_id)
schema_version: version PK
```

### 2. `database/manager.py` — адаптер DatabaseManager

Полная обратная совместимость со старым API:

- `data` (property) — кэш в памяти для синхронного доступа (`monitor.py`, `healthcheck.py`)
- `get_user_data(uid)` — как и раньше, создаёт запись по умолчанию
- `update_user()`, `add_patient()`, `toggle_monitoring()`, `delete_patient()` и т.д. — пишут сразу в SQLite
- Произвольные поля (например, "name") сохраняются в `extra`

### 3. `database/doctor_manager.py` — адаптер DoctorManager

- `load()` — загружает всех врачей из SQLite в `self.data`
- `merge_doctors()` — делегирует в `Database.merge_doctors()`

## Часть 2. Ревью SQLite-реализации (вечерняя сессия)

**Результат:** 64/64 тестов проходят, 0 Pylance-ошибок, устранено 8 проблем безопасности и качества

### Проведённый анализ

Выполнен детальный обзор всей SQLite-реализации: `database/database.py`, `database/manager.py`, `database/doctor_manager.py`, `tests/`, `services/monitor.py`, `services/healthcheck.py`, `handlers/`, `config.py`, `main.py`.

**Найдено 8 проблем, все исправлены:**

| # | Проблема | Ранее | Стало |
|---|---------|-------|-------|
| P1 | `update_user()` — поля пишутся без единой транзакции | 6 отдельных `update_user_field()` + `set_last_notification_id()` + `update_extra()`, каждый со своим `.commit()` — при mid-air сбое данные частично сохранялись | Все UPDATE обёрнуты в `BEGIN` / `COMMIT` / `ROLLBACK` через прямой доступ к `self._db._conn` |
| P2 | Потенциальная SQL-инъекция в `update_user_field()` | `f"UPDATE users SET {field} = ?"` — имя поля подставлялось через f-string | Добавлен whitelist `_ALLOWED_FIELDS` (`frozenset`), невалидное поле → `ValueError` |
| P3 | `data` возвращает mutable-ссылку на кэш | Внешний код мог прямо мутировать `data[uid]["patients"]["p1"]` — изменения не попадали в SQLite | `data` возвращает `copy.deepcopy(self._data_cache)` |
| P4 | `get_user_data()` создаёт структуру только в памяти | Если uid нет в SQLite, а handler пишет напрямую в `data` — данные теряются | Косвенно исправлено P3 + рекомендация использовать `update_user()` |
| P5 | `shallow copy` в `monitor_loop()` | `db.data.copy()` копировал только корневой dict — вложенные `monitoring`, `patients` жили в оригинале | Убран `.copy()`, т.к. `db.data` теперь сам возвращает deepcopy |
| P7 | `assert c is not None` — отключается при `-O` | 15 мест с `assert c is not None` в `database.py` | Все заменены на `if c is None: raise RuntimeError(...)` |
| P8 | Закомментированный код | `# import sqlite3 # Удален` | Удалён |
| P9 | Сбивающий INSERT в `ensure_user()` | `VALUES (?, '{}', '{}', '{}')` с одним параметром `(uid,)` — неочевидно | `INSERT OR IGNORE INTO users (uid) VALUES (?)` — явно и чисто |
