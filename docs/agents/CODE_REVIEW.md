## 8. Utils (`utils/`)

### 8.1 `utils/cache.py`

- [x] `swap_cache_key` — атомарная read-compare-write под lock
- [x] `delete_cache_keys_by_prefix` — массовое удаление

**Проблемы:**

- **⚠️ `spam_cache = TTLCache(maxsize=1000, ttl=1.0)`** — жёсткие значения, не из конфига. maxsize=1000 может быть переполнен при большом количестве пользователей. TTL=1.0 секунда может не защитить от реального спама.
- **⚠️ Файловый кэш на каждый запрос** — при активном мониторинге (сотни врачей) каждые ~60 секунд будет读写 JSON-файла. На Windows это может быть медленно. Лучше in-memory cache с периодической синхронизацией.
- **⚠️ `swap_cache_key()` — кэш не иммутабелен по old_value** (строка 36-38). Если old_value — список, и кто-то его мутирует после получения, но до записи — данные в кэше могут быть повреждены. На практике не происходит из-за lock.

- [x] Middleware регистрируется как outer_middleware

**Проблемы:**

- **🔴 `load_config_from_db(database)` вызывается **после** `sync_clinic_names()`** (строки 58-65). Если в БД переопределён `API_BASE_URL`, то `sync_clinic_names()` использует старый (из .env). Это баг — загрузка конфигов должна быть раньше.

- **⚠️ `background_tasks` содержит как корутины, созданные через `create_task`, так и прямые вызовы** — все корректны.
- **⚠️ В `finally` блоке (строка 131-139) второй вызов `bot.session.close()`** (строка 139). `dp.start_polling()` уже закрывает сессию при остановке. Двойное закрытие может вызвать исключение (но оно проглатывается).

## Итого по проекту

**Общее качество кода:** хорошее. Проект структурирован, используется современный Python (async/await, Pydantic v2, aiogram 3.x). Есть миграции БД, rate limiting, тесты, graceful shutdown, интеграция с Sentry/NTFY.

**Основные риски:**

1. Нарушение инкапсуляции `_conn` — при рефакторинге Database сломаются менеджеры
2. Порядок инициализации в main.py — конфиги из БД не применяются к sync_clinic_names
3. Хардкоды CSRF_TOKEN, DEFAULT_CLINIC_ID, DENTAL_CLINIC_ID — при изменении на портале всё сломается
4. Файловый кэш может стать узким местом при масштабировании
