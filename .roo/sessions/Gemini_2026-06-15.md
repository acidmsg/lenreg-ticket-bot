## 1. Изоляция секретов (Security First)

Секреты никогда не должны попадать в историю Git, независимо от типа репозитория и выбранной архитектуры.

- Токены Telegram-ботов, пароли к базам данных и ключи API строго выносим в файл `.env`.
- Файл `.env` обязательно добавляется в `.gitignore`.
- В репозитории сохраняется только `.env.example` с пустыми значениями для демонстрации структуры.

## 2. Архитектурные подходы

В зависимости от конечной цели, логику разделяют двумя способами.

### Вариант А: Два репозитория (Рекомендуемый)

Оптимальный подход, если служебные файлы (например, промпты для систем размышления ИИ, инструкции для Roo Code/Cline, черновые скрипты) составляют весомую часть проекта и должны быть гарантированно скрыты от посторонних глаз.

- **Private Repo (Source of Truth):** Приватный репозиторий. Здесь ведется вся разработка, лежат тесты, инструменты автоматизации и сырые данные.
- **Public Repo (Production/Release):** Открытый репозиторий, содержащий исключительно чистый рабочий код.

**Синхронизация через GitHub Actions:**
Настраивается автоматический CI/CD пайплайн. При пуше в главную ветку или создании релиза, GitHub Action берет только нужную директорию (например, `src/`) и пушит её в публичный репозиторий.

```yaml
# .github/workflows/push-to-public.yml
name: Push Production Code
on:
  push:
    branches:
      - main
jobs:
  build-and-push:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - name: Push to Public Repo
        uses: cpina/github-action-push-to-another-repository@main
        env:
          API_TOKEN_GITHUB: ${{ secrets.PUBLIC_REPO_TOKEN }}
        with:
          source-directory: 'src' # Export only production code
          destination-github-username: 'your-username'
          destination-repository-name: 'public-repo'
          target-branch: 'main'
```

### Вариант Б: Изолированные артефакты (Монорепозиторий)

Подходит, если исходный код проекта в GitHub остается закрытым для публики, а разделение нужно только для того, чтобы на сервер не летел мусор.

- Для деплоя собирается Docker-образ.
- В файл `.dockerignore` вписываются все локальные утилиты, markdown-заметки и логи.

```text
# .dockerignore
docs/
ai_prompts/
tests/
*.log
docker-compose.yml
```

В результате на сервер отправляется минималистичный и безопасный контейнер, в то время как вся кухня разработки остается версионированной в GitHub.

## 3. Точечное исключение исходников (.gitattributes)

Если код распространяется публично через вкладку **Releases** (скачиваемые архивы `.zip` и `.tar.gz`), используйте встроенный механизм `export-ignore`.

1. Создайте файл `.gitattributes` в корне проекта.
2. Перечислите пути, которые должны игнорироваться при архивации релиза GitHub.

```text
# .gitattributes
.github/ export-ignore
tests/ export-ignore
scripts/ export-ignore
.gitignore export-ignore
```

При скачивании исходников из релиза, пользователи получат чистый код без этих папок, хотя в самом репозитории они будут продолжать лежать и версионироваться.

Единственный архитектурно верный подход в таком случае — **приватный монорепозиторий с автоматической CI/CD дистрибуцией**.

Главная причина: вы храните абсолютно всё (код, докер, служебные скрипты, промпты для AI-агентов) в одном закрытом месте, где ведется история версий. Наружу (в публичный код и на сервер) попадает только отфильтрованный срез данных через GitHub Actions.

## 1. Структура монорепозитория

Вся разработка ведется в приватном репозитории. Служебная инфраструктура физически отделена от боевого кода на уровне директорий.

```text
my-private-bot/
├── .cline/                 # Служебное: контекст и память AI-агента (остается в привате)
├── .roo/                   # Служебное: настройки Roo Code (остается в привате)
├── prompts/                # Служебное: системные промпты и черновики (остается в привате)
├── src/                    # Боевое: исходный код (публикуется)
│   ├── main.py
│   └── requirements.txt
├── .dockerignore           # Боевое: правила фильтрации для контейнера
├── Dockerfile              # Боевое: инструкции сборки
└── docker-compose.yml      # Боевое: запуск на сервере
```

## 2. Изоляция Docker-образа

Чтобы служебная инфраструктура не улетела на VPS внутри контейнера, она жестко отсекается на этапе сборки.

В `.dockerignore` вносятся все служебные директории:

```text
# Игнорируем инфраструктуру разработки
.cline/
.roo/
prompts/
scripts/

# Игнорируем Git и локальные файлы
.git/
.env
*.md
```

При выполнении `docker build` демон Docker даже не узнает о существовании этих файлов. Контейнер останется легковесным и безопасным.

## 3. Автоматическая дистрибуция (GitHub Actions)

Настраивается Workflow, который при создании релиза (или пуше в `main`) выполняет две независимые задачи:

1. Собирает Docker-образ и пушит его в GitHub Container Registry (GHCR) или Docker Hub.
2. Копирует только разрешенные файлы (исходники + докер) и пушит их в публичный репозиторий.

Пример рабочего пайплайна `.github/workflows/release.yml`:

```yaml
name: Release Pipeline

on:
  release:
    types: [published]

jobs:
  # Задача 1: Синхронизация кода в публичный репозиторий
  sync_code:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout private repo
        uses: actions/checkout@v4

      - name: Prepare public release directory
        run: |
          # Создаем временную директорию для чистого кода
          mkdir -p public_release
          # Копируем только нужное
          cp -r src/ public_release/
          cp Dockerfile public_release/
          cp docker-compose.yml public_release/
          cp README.md public_release/

      - name: Push to Public Repository
        uses: cpina/github-action-push-to-another-repository@main
        env:
          # Токен доступа (хранить в GitHub Secrets приватного репозитория)
          API_TOKEN_GITHUB: ${{ secrets.PUBLIC_REPO_TOKEN }}
        with:
          source-directory: 'public_release'
          destination-github-username: 'your-username'
          destination-repository-name: 'public-bot-repo'
          target-branch: 'main'

  # Задача 2: Сборка и публикация Docker-образа
  build_docker:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout private repo
        uses: actions/checkout@v4

      - name: Login to GitHub Container Registry
        uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - name: Build and push Docker image
        uses: docker/build-push-action@v5
        with:
          context: .
          push: true
          # Тег образа будет соответствовать версии релиза
          tags: ghcr.io/your-username/your-bot:${{ github.event.release.tag_name }},ghcr.io/your-username/your-bot:latest
```

## Аналитический вывод (1)

Данный подход полностью решает проблему:

- **Версионирование:** 100% файлов (включая служебные) версионируются в закрытом приватном репозитории.
- **Чистота кода:** Публичный репозиторий содержит только код и файлы развертывания, без следов вашей локальной инфраструктуры или инструкций для AI.
- **Безопасность продакшена:** Docker-контейнер собирается только из необходимых исходников благодаря `.dockerignore`, исключая утечку служебных скриптов на конечный сервер.

Да, физически на GitHub это будет **два отдельных репозитория**.

Термин «монорепозиторий» в моем предыдущем ответе означает, что локально для вас это один проект. Вы не дробите исходники и инфраструктуру по разным папкам на своем компьютере, а ведете всю работу в единой кодовой базе.

Вот как распределяются роли между этими двумя репозиториями:

### 1. Основной репозиторий (Приватный)

Это ваш единственный рабочий инструмент (Source of Truth). Вы клонируете его на свой ПК и делаете все коммиты **только сюда**.

- **Статус:** `Private` (полностью закрыт от посторонних).
- **Содержимое:** 100% файлов проекта (весь код + `.cline`, `.roo`, `prompts`, скрипты автоматизации).
- **Ваша роль:** Писать код, тестировать, создавать релизы (Git Tags).

### 2. Репозиторий-витрина (Публичный)

Это автоматизированное зеркало для чистого кода. Вы **никогда** не клонируете его для локальной разработки и не делаете в него коммиты вручную.

- **Статус:** `Public` (открыт для сообщества).
- **Содержимое:** Только отфильтрованная выжимка (например, папка `src/`, `Dockerfile`, `docker-compose.yml` и `README.md`).
- **Ваша роль:** Отсутствует. Репозиторий живет своей жизнью и обновляется роботом.

### Связующее звено (CI/CD)

Мостом между ними служит GitHub Actions. Как только вы создаете новый релиз в **Приватном репозитории**, скрипт на серверах GitHub выполняет грязную работу:

1. Создает временную пустую директорию.
2. Копирует туда только разрешенные файлы из Приватного репозитория (оставляя служебный мусор за бортом).
3. Автоматически делает `git push` в **Публичный репозиторий** с помощью токена доступа.

## Аналитический вывод (4)

С точки зрения архитектуры GitHub — это два разных репозитория. С точки зрения процесса разработки — это один проект. Вы работаете в едином закрытом пространстве, полностью контролируя все версии и AI-промпты, а наличие второго репозитория служит исключительно целям безопасной публичной дистрибуции чистого кода без ручного вмешательства.

```yaml
name: Production Release Automation

on:
  release:
    types: [published] # Запуск только при публикации нового релиза

jobs:
  # Задача 1: Синхронизация кода в публичный репозиторий
  sync_to_public:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout private repository
        uses: actions/checkout@v4

      - name: Validate project structure
        run: |
          if [ ! -d "src" ]; then
            echo "Критическая ошибка: Директория src/ не найдена."
            exit 1
          fi # Базовая проверка безопасности перед сборкой

      - name: Prepare clean release directory
        run: |
          mkdir -p clean_release
          # Копируем только разрешенные компоненты
          cp -r src clean_release/
          cp Dockerfile clean_release/
          cp docker-compose.yml clean_release/
          cp README.md clean_release/

      - name: Push to Public Repository
        uses: cpina/github-action-push-to-another-repository@main
        env:
          API_TOKEN_GITHUB: ${{ secrets.PUBLIC_REPO_TOKEN }}
        with:
          source-directory: 'clean_release'
          destination-github-username: 'your-github-username' # Заменить на ваш логин
          destination-repository-name: 'your-public-repo-name' # Заменить на имя публичного репо
          target-branch: 'main'
          commit-message: 'Release ${{ github.event.release.tag_name }}'

  # Задача 2: Сборка и публикация Docker-образа
  build_and_push_docker:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout private repository
        uses: actions/checkout@v4

      - name: Set up Docker Buildx
        uses: docker/setup-buildx-action@v3 # Настройка сборщика

      - name: Login to GitHub Container Registry
        uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }} # Автоматический токен GitHub Actions

      - name: Build and push Docker image
        uses: docker/build-push-action@v5
        with:
          context: . # Контекст сборки (файлы отсекутся через .dockerignore)
          push: true
          tags: |
            ghcr.io/${{ github.repository_owner }}/prod-app:${{ github.event.release.tag_name }}
            ghcr.io/${{ github.repository_owner }}/prod-app:latest
```

## Стратегии именования

| Подход                             | Приватный (Рабочий)                   | Публичный (Релизный)                 | Комментарий                                                                               |
| :--------------------------------- | :------------------------------------ | :----------------------------------- | :---------------------------------------------------------------------------------------- |
| **Релизный фокус (Рекомендуемый)** | `project-core` или `project-internal` | `project`                            | Публичный репозиторий получает чистое имя. Рабочий помечается как исходное ядро.          |
| **Фокус на разработке**            | `project`                             | `project-release` или `project-dist` | Рабочий репозиторий сохраняет основное имя. Публичный выступает только как точка раздачи. |

## Примеры применения

Если ваш проект — это скрипт или Telegram-бот для автоматизации, используйте логику ядра:

- **Приватный репозиторий (Где кипит работа):** `github.com/acid_grip/automation-bot-core`
- **Публичный репозиторий (Витрина для людей):** `github.com/acid_grip/automation-bot`

В экшене `.github/workflows/release.yml` приватного репозитория переменная назначения будет выглядеть так:

```yaml
destination-github-username: 'acid_grip'
destination-repository-name: 'automation-bot' # Имя публичного репо
```
