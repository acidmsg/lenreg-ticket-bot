# Правила работы с .env и .env.example

## Назначение файлов

| Файл | Содержимое | Git |
|------|-----------|-----|
| `.env` | Реальные данные (токены, пароли, URL) | **В .gitignore** — никому не показывать |
| `.env.example` | **Только реальные** публичные/нечувствительные значения; для секретов — плейсхолдеры | Коммитится |

## Правила синхронизации

1. `.env.example` — **образец**, по которому разработчик создаёт свой `.env`.
2. Все **нечувствительные** ключи должны быть синхронизированы между `.env` и `.env.example` (одинаковые ключи, одинаковые значения).
3. **Чувствительные** ключи в `.env.example` заменяются на плейсхолдеры вида `your_..._here`.

## Чувствительные ключи (должны быть плейсхолдерами в .env.example)

- `BOT_TOKEN`
- `PROXY_URL` (если содержит пароль/адрес)
- `ADMIN_IDS` (персональные Telegram ID)
- `DISCOVERY_PATIENT_ID_ADULT` (ID пациента на портале)
- `DISCOVERY_PATIENT_ID_CHILD` (ID пациента на портале)

## Нечувствительные ключи (реальные значения в .env.example)

- `SQLITE_DB_PATH`
- `CACHE_PATH`
- `PYTHONUTF8`
- Все ключи из секции `# === Настройки API ===` и т.п.

## При добавлении нового ключа

1. Добавить ключ с реальным значением в `.env`.
2. Добавить ключ в `.env.example`:
   - Если чувствительный — плейсхолдер `your_..._here`.
   - Если публичный — реальное значение.
3. Убедиться, что `pydantic_settings` в `config.py` подхватывает новый ключ.
