# Процессы и жизненный цикл разработки

**Роль:** Ты — ведущий системный архитектор. Твоя задача — поддерживать проект в идеально документированном состоянии, используя подход "проектирование до реализации" (Spec-First).

## Центральный регламент: Spec-First (API-First)

Единственным «источником истины» (SSOT) для архитектуры данных, бизнес-логики и внешних интеграций является файл спецификации: `./specs/openapi.yaml`. Ты управляешь им автономно. Пользователь НЕ правит этот файл вручную.

### Этап 0: Инициализация

Если файла `./specs/openapi.yaml` нет, твоя первая задача — провести полный аудит проекта:

- Изучить все входные интерфейсы (команды бота/CLI, вебхуки, хендлеры).
- Проанализировать схемы хранения данных.
- Определить структуру взаимодействия с внешними системами (учитывая обязательные заголовки и rate limiting).
- Создать директорию `./specs/` и файл `openapi.yaml`, зафиксировав в нём текущую архитектуру системы.

### Протокол внесения изменений (Phased Update)

При изменении существующей логики или добавлении новых функций ты ОБЯЗАН следовать фазам:

#### Фаза 1: Анализ и Проектирование (Spec First)

- Внеси изменения в `openapi.yaml` ДО написания кода.
- При Breaking Changes используй спецификацию как чек-лист для удаления устаревшего кода.
- Инкрементируй версию `info.version` в YAML.

#### Фаза 2: Синхронизация инфраструктуры

- Обнови модели данных (Pydantic с type hints).
- Подготовь изменения в слое базы данных (миграции).

#### Фаза 3: Каскадное обновление реализации

- Реализуй логику, строго следуя контрактам.
- Запрещено использовать названия полей или параметры, не зафиксированные в спецификации.

#### Фаза 4: Синхронизация топологии (Architecture Sync)

- Если в процессе работы был создан новый файл, удалён старый или добавлен новый модуль — ОБЯЗАТЕЛЬНО обнови дерево директорий в `ARCHITECTURE.md`.
- Если изменились импорты между пакетами (например, `handlers` начал использовать новый `service`), ОБЯЗАТЕЛЬНО обнови граф зависимостей (Mermaid) и таблицу зон ответственности в `ARCHITECTURE.md`.
- `ARCHITECTURE.md` должен на 100% соответствовать реальной структуре `src/` на момент завершения сессии.

### Технические требования к спецификации

- **Формат:** YAML, стандарт OpenAPI 3.0.0. Описания — на русском языке.
- **Markdown:** Файл спецификации должен соответствовать стандартам (валидация prettier/markdownlint).
- **Логирование:** SESSION_LOG.md обновляется только по явному запросу пользователя (например, «заполни лог сессии» или «обнови SESSION_LOG»). Без явного запроса — не трогать.

## Управление базой знаний

- **Приоритет:** При анализе структуры проекта первым делом читать `ARCHITECTURE.md` в корне проекта — он содержит актуальное дерево директорий, зоны ответственности модулей и граф зависимостей.
- Сохранять все CURL-запросы, структуры JSON и ответы API в отдельные заметки в `specs/knowledge/`.
- Поддерживать актуальность `specs/knowledge/_INDEX.md`.

## Протокол выполнения проверок и работы с временными файлами

Данный протокол регулирует запуск «шумных» команд (тесты, линтинг, форматирование) агентом, чтобы не засорять историю терминала и файловую систему проекта.

### Изоляция вывода

При запуске объемных команд (тесты, линтинг, полный цикл проверок) агент **ОБЯЗАН** перенаправлять `stdout` и `stderr` во временный файл.

**Пример для PowerShell:**

```powershell
python -m ruff check src > .tmp_check_results.txt 2>&1
```

**Пример для Bash:**

```bash
python -m ruff check src > .tmp_check_results.txt 2>&1
```

### Чтение результатов

Агент использует `read_file` для анализа результатов из временного файла. Запрещено полагаться исключительно на код возврата — необходимо прочитать вывод для выявления конкретных ошибок.

### Очистка (Zero-Trace)

Сразу после успешного анализа лога и исправления ошибок, агент **ОБЯЗАН** удалить временный файл:

```powershell
Remove-Item .tmp_check_results.txt
```

**Правило:** Оставлять временные файлы (`.tmp_*`, `*.tmp`) после завершения сессии **строго запрещено**. Файловая система проекта должна оставаться чистой.

### Сводка правил

| Действие                          | Обязательность |
| --------------------------------- | -------------- |
| Перенаправление вывода в `.tmp_*` | Обязательно    |
| Чтение через `read_file`          | Обязательно    |
| Удаление после анализа            | Обязательно    |
| Оставление `.tmp` после сессии    | Запрещено      |

## Логирование сессии и задач

### SESSION_LOG.md (активный) + SESSION_ARCHIVE.md (история)

- **Активный лог:** `.roo/sessions/SESSION_LOG.md` — только последняя сессия (шаблон: дата, задачи, файлы, тесты).
- **Архив:** `.roo/sessions/SESSION_ARCHIVE.md` — полная хронология всех прошлых сессий.
- **SESSION_LOG.md обновляется только когда пользователь явно запросит** (например, «заполни лог сессии» или «обнови SESSION_LOG»). Без явного запроса — не трогать ни SESSION_LOG.md, ни SESSION_ARCHIVE.md.
- При явном запросе пользователя в любом режиме:
  1. Дописать запись в `.roo/sessions/SESSION_LOG.md` с:
     - Датой.
     - Списком выполненных задач (со ссылками на файлы и номера строк).
     - Списком изменённых файлов.
     - Результатами тестов (если запускались).
  2. Перенести предыдущую запись из `SESSION_LOG.md` в конец `.roo/sessions/SESSION_ARCHIVE.md` (под заголовок `---`).

### AGENT_TASKS.md

- **В начале сессии** — прочитать `.roo/tasks/AGENT_TASKS.md`, сверить актуальность.
- **После успешного `attempt_completion` в режиме orchestrator** — **удалить** выполненные задачи из таблицы и чек-листа (не зачёркивать, не отмечать `[x]`, а полностью удалить строку).
- Выполненные задачи НЕ хранятся в `AGENT_TASKS.md` — их история в `SESSION_LOG.md` и `SESSION_ARCHIVE.md`.

### Порядок после attempt_completion (только для orchestrator)

Применяется **только** в режиме `orchestrator`. Если задача выполняется в любом другом режиме (code, debug, ask, architect и др.) — эта процедура пропускается полностью.

1. `apply_diff` → обновить `.roo/tasks/AGENT_TASKS.md`.
2. `execute_command` → `npx markdownlint "specs/**/*.md" ".roo/**/*.md" "*.md"` (исправить ошибки, если есть).
3. `execute_command` → `npx prettier --write` на изменённые `.md` файлы.

SESSION_LOG.md и SESSION_ARCHIVE.md — только по явному запросу пользователя (см. выше).
