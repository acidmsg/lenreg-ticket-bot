# Технические стандарты: Python и Markdown

Применяются ко **всем** текущим и будущим задачам в проекте.

## Философия кода

**Базовые принципы, управляющие каждым изменением кода в проекте.**

### Ⅰ Модульность — код строится из переиспользуемых блоков

- **Базовый компонент + фабрика-обёртка.** Каждую переиспользуемую единицу (форма, таблица, календарь) пиши один раз как общий строительный блок. Для конкретных сценариев создавай тонкую фабрику-обёртку, которая передаёт правильные настройки. Запрещено копировать код блока и менять одну-две настройки.
- **Одна функция — одно дело.** Каждая функция/метод выполняет ровно одну задачу. Если функция переваливает за ~30 строк или делает несколько несвязанных вещей — разбей на несколько.
- **Бойлерплейт выносится.** Повторяющаяся обвязка (например, `cast(Jinja2Templates, ...)` в каждом роутере, проверки прав, логирование) выносится в хелпер/утилиту.

### Ⅱ Чистота — кода ровно столько, сколько нужно

- **Избыточная проверка = мусор.** Проверка избыточна, если сбой в этом месте невозможен из-за гарантий выше по коду. Проверка необходима, если данные приходят извне (API, пользователь, файл). Не пиши `if x is None:` повторно там, где выше уже проверено.
- **Единый источник данных.** Списки, настройки, конфигурации полей — живут в одном месте. Запрещено дублировать одни и те же данные (список специальностей, перечень городов, настройки полей) в разных файлах.

### Ⅲ Эволюция — рефакторинг при расширении

- **Сначала оглянись, потом пиши.** Прежде чем писать новую функцию — проверь: нет ли в проекте похожей? Нельзя ли её обобщить?
- **Рефакторинг ДО расширения.** При добавлении новой сущности (например, новой формы) сначала отрефактори существующий код так, чтобы он стал универсальным, и только потом добавляй новую сущность на его основе. Запрещено копировать старую функцию и править в ней пару строк — обобщай старую.

### Ⅳ Стиль — имена говорят сами за себя

- **Функция = глагол, переменная = существительное.** `get_user()`, `save_data()`, `user_name`, `doctor_list`.
- **Полные имена.** Никаких сокращений вроде `p_dt(d)` — только `process_doctor_data(doctor_info)`. Краткие имена (`i`, `n`, `x`) допустимы только для счётчиков циклов.
- **Код — на английском.** Имена переменных, функций, классов — строго на английском. Русский — только в комментариях и docstring.

## Python (.py)

### Типизация

- Обязательные type hints для **всех** сигнатур функций и методов.
- Отсутствие ошибок mypy (`strict`-опционально, но `--check-untyped-defs` минимум).
- Аннотации для переменных уровня модуля.

### Форматирование

- Жёсткое форматирование через Ruff (линтер + форматтер) и Black-совместимый стиль.
- Конфигурация Ruff в `pyproject.toml` задаёт стиль кавычек (`"`), отступы (4 пробела), целевую версию Python.

### Импорты

- **Абсолютные** предпочтительнее относительных.
- Префикс `src.` для всех внутренних импортов (например, `from src.config import settings`).
- Строгая сортировка: стандартная библиотека → сторонние → внутренние (isort-совместимо, через Ruff).

### Прочее

- PEP-8: 4 пробела, 100 символов на строку, snake_case для функций/переменных, PascalCase для классов.
- Докстринги: тройные двойные кавычки `"""`, первая строка — краткое описание.

## Обработка ошибок

- Явные проверки на `None` для объектов, которые могут быть `None` (например, `message.from_user` или `call.message` в aiogram).
- Различать типы исключений (не голый `except Exception`).
- Всегда логировать трассировку при перехвате ошибок API.

## Взаимодействие с внешними API

- Все запросы включают обязательные заголовки (например, `X-Requested-With`, `Content-Type`, `X-CSRFToken`, `Cookie`, `Origin`, `Referer`).
- Rate limiting через `aiolimiter` — отдельные лимитеры для разных типов задач (мониторинг, discovery, healthcheck, пользовательские запросы).
- Retry для методов API: 3 попытки с задержкой 2с на 5xx и сетевые ошибки.

## Контроль соответствия схем API (API Drift Prevention)

- Pydantic-модели в `src/api/models.py` являются единственной реализацией схем данных, определённых в `docs/openapi.yaml`.
- При любом изменении структуры ответа API (добавление/удаление/переименование поля) обязательная последовательность:
  1. Обновить спецификацию `docs/openapi.yaml` (Spec-First).
  2. Обновить Pydantic-модель в `src/api/models.py`.
  3. Пересгенерировать JSON-схему в `docs/schemas/` командой `python scripts/generate_api_schemas.py`.
  4. Убедиться, что тесты в `tests/api/` проходят без ошибок.
- Запрещено добавлять поля в Pydantic-модели, не зафиксированные в `docs/openapi.yaml`.
- При обнаружении расхождения между `docs/schemas/` и `src/api/models.py` задача фиксируется в `docs/agents/TECH_DEBT.md` как `Severity: Critical`.

## Отказоустойчивость Redis (Graceful Degradation)

- Все обращения к Redis обязаны перехватывать `aioredis.RedisError`, `ConnectionError` и `TimeoutError`.
- Стратегия деградации по компонентам:
  - **FSM-хранилище:** при недоступности Redis — логировать ошибку, уведомить пользователя о временной недоступности, не крашить бот.
  - **Кэш мониторинга:** при ошибке Redis — пропустить итерацию мониторинга, залогировать с уровнем `WARNING`.
  - **Rate limiting (Redis Sorted Sets):** при ошибке Redis — пропустить проверку лимита (fail-open), залогировать с уровнем `WARNING`.
  - **Spam-защита (SET NX EX):** при ошибке Redis — пропустить проверку (fail-open).
- Логировать каждый случай деградации с контекстом: компонент, причина ошибки, действие по умолчанию.
- Healthcheck (`src/services/healthcheck.py`) обязан проверять доступность Redis и отражать статус в отчёте.

## Rate Limiting при отправке уведомлений в Telegram

- Telegram Bot API имеет жёсткие лимиты: не более 30 сообщений в секунду суммарно, не более 1 сообщения в секунду в один чат (личка).
- При массовой рассылке уведомлений (например, `src/services/monitor.py`) обязательно использовать `aiolimiter.AsyncLimiter` с лимитом не более **25 сообщений/сек** суммарно.
- Ошибка `429 Too Many Requests` от Telegram API обрабатывается отдельно: извлекается `retry_after` из ответа, отправка откладывается на указанное время через `asyncio.sleep(retry_after)`.
- Запрещено использовать `asyncio.gather()` без ограничения конкурентности при параллельной отправке уведомлениям множеству пользователей — использовать `asyncio.Semaphore` или очередь.

## Конкурентность

- `asyncio.Lock` для атомарных операций с разделяемыми данными.
- `aiofiles` для асинхронного файлового I/O.
- `asyncio.Task` для фоновых задач; корректная отмена через `task.cancel()` + `asyncio.gather(return_exceptions=True)`.

## Хранение данных

- Runtime-данные (SQLite БД, файловый кэш) — в `data/`.
- Слой базы данных: ядро (SQLite WAL-режим, миграции) + адаптер с in-memory кэшем + специализированные кэш-менеджеры.

## Конфигурация

- Конфиденциальные данные — только в `.env`, не хардкодить.
- Нечувствительные параметры — в БД (таблица `config`) с переопределением через `load_config_from_db()`.
- Двухуровневое переопределение: `.env` → Settings (pydantic) → БД.

## Тестирование

- Временные SQLite-файлы в `tests/test_data/`, очистка после сессии.
- Моки через `unittest.mock` и monkeypatch.
- `gc.collect()` в фикстурах для контроля памяти.

## Правила работы с .env и .env.example

### Назначение файлов

| Файл           | Содержимое                                                                           | Git                                     |
| -------------- | ------------------------------------------------------------------------------------ | --------------------------------------- |
| `.env`         | Реальные данные (токены, пароли, URL)                                                | **В .gitignore** — никому не показывать |
| `.env.example` | **Только реальные** публичные/нечувствительные значения; для секретов — плейсхолдеры | Коммитится                              |

### Правила синхронизации

1. `.env.example` — **образец**, по которому разработчик создаёт свой `.env`.
2. Все **нечувствительные** ключи должны быть синхронизированы между `.env` и `.env.example` (одинаковые ключи, одинаковые значения).
3. **Чувствительные** ключи в `.env.example` заменяются на плейсхолдеры вида `your_..._here`.

### Категоризация ключей

**Чувствительные ключи** (должны быть плейсхолдерами в `.env.example`):

- `BOT_TOKEN`, `API_TOKEN`, `SECRET_KEY`
- `PROXY_URL` (если содержит пароль/адрес)
- `ADMIN_IDS` (персональные ID пользователей)
- `DATABASE_URL` (если содержит пароль)
- Любые ключи доступа к внешним сервисам (Sentry DSN, NTFY URL, etc.)

**Нечувствительные ключи** (реальные значения в `.env.example`):

- `SQLITE_DB_PATH`, `CACHE_PATH`
- `PYTHONUTF8`
- `LOG_LEVEL`, `ENVIRONMENT`
- Таймауты, интервалы, пороговые значения

### При добавлении нового ключа

1. Добавить ключ с реальным значением в `.env`.
2. Добавить ключ в `.env.example`:
   - Если чувствительный — плейсхолдер `your_..._here`.
   - Если публичный — реальное значение.
3. Убедиться, что `pydantic_settings` в `config.py` подхватывает новый ключ.

## Markdown (.md)

### Иерархия заголовков

- **Один H1** (`#`) на файл.
- Строго последовательная иерархия: H1 → H2 → H3 (без пропусков уровней).

### Блоки кода

- Обязательное указание языка: `python`, `json`, `bash`, `text`, `mermaid`, `yaml`.

### Списки

- Единый маркер: `-`.
- Пустая строка перед и после списка.

### Пробелы

- Отсутствие висячих пробелов в конце строк.
- Завершение файла ровно одной пустой строкой (`\n` в конце).

### Валидация

- Файлы должны проходить `markdownlint` и `prettier`.
- Форматирование через `npx prettier --write` (proseWrap: preserve, printWidth: 120).
- Проверка markdownlint: `npx markdownlint "docs/**/*.md" ".roo/**/*.md" "*.md"` (ожидается 0 errors).

## Стилистика текста

- Лаконичность, сухость, технический стиль.
- Исключены: эмоции, избыточные описания, вода.
- Только факты, алгоритмы, спецификации.

## Ссылки на файлы

- В markdown-документах ссылки на файлы оформляются с указанием строки: ``[`filename.ext:line`](relative/path/file.ext:line)``.
- Для конструкций кода указывается и строка: ``[`function_name()`](path/file.py:42)``.
