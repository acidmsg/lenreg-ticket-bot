# Технические стандарты: Python и Markdown

Применяются ко **всем** текущим и будущим задачам в проекте.

---

## Python (.py)

### Типизация

- Обязательные type hints для **всех** сигнатур функций и методов.
- Отсутствие ошибок mypy (`strict`-опционально, но `--check-untyped-defs` минимум).
- Аннотации для переменных уровня модуля.

### Форматирование

- Жёсткое форматирование через Ruff (линтер + форматтер) и Black-совместимый стиль.
- Конфигурация Ruff в `pyproject.toml` задаёт стиль кавычек (`"`), отступы (4 пробела), целевую версию Python.

### Импорты

- **Абсолютные** предпочтительнее относительных.
- Префикс `src.` для всех внутренних импортов (например, `from src.config import settings`).
- Строгая сортировка: стандартная библиотека → сторонние → внутренние (isort-совместимо, через Ruff).

### Прочее

- PEP-8: 4 пробела, 100 символов на строку, snake_case для функций/переменных, PascalCase для классов.
- Докстринги: тройные двойные кавычки `"""`, первая строка — краткое описание.

## Обработка ошибок

- Явные проверки на `None` для объектов, которые могут быть `None` (например, `message.from_user` или `call.message` в aiogram).
- Различать типы исключений (не голый `except Exception`).
- Всегда логировать трассировку при перехвате ошибок API.

## Взаимодействие с внешними API

- Все запросы включают обязательные заголовки (например, `X-Requested-With`, `Content-Type`, `X-CSRFToken`, `Cookie`, `Origin`, `Referer`).
- Rate limiting через `aiolimiter` — отдельные лимитеры для разных типов задач (мониторинг, discovery, healthcheck, пользовательские запросы).
- Retry для методов API: 3 попытки с задержкой 2с на 5xx и сетевые ошибки.

## Конкурентность

- `asyncio.Lock` для атомарных операций с разделяемыми данными.
- `aiofiles` для асинхронного файлового I/O.
- `asyncio.Task` для фоновых задач; корректная отмена через `task.cancel()` + `asyncio.gather(return_exceptions=True)`.

## Хранение данных

- Runtime-данные (SQLite БД, файловый кэш) — в `data/`.
- Слой базы данных: ядро (SQLite WAL-режим, миграции) + адаптер с in-memory кэшем + специализированные кэш-менеджеры.

## Конфигурация

- Конфиденциальные данные — только в `.env`, не хардкодить.
- Нечувствительные параметры — в БД (таблица `config`) с переопределением через `load_config_from_db()`.
- Двухуровневое переопределение: `.env` → Settings (pydantic) → БД.

## Тестирование

- Временные SQLite-файлы в `tests/test_data/`, очистка после сессии.
- Моки через `unittest.mock` и monkeypatch.
- `gc.collect()` в фикстурах для контроля памяти.

## Правила работы с .env и .env.example

### Назначение файлов

| Файл           | Содержимое                                                                           | Git                                     |
| -------------- | ------------------------------------------------------------------------------------ | --------------------------------------- |
| `.env`         | Реальные данные (токены, пароли, URL)                                                | **В .gitignore** — никому не показывать |
| `.env.example` | **Только реальные** публичные/нечувствительные значения; для секретов — плейсхолдеры | Коммитится                              |

### Правила синхронизации

1. `.env.example` — **образец**, по которому разработчик создаёт свой `.env`.
2. Все **нечувствительные** ключи должны быть синхронизированы между `.env` и `.env.example` (одинаковые ключи, одинаковые значения).
3. **Чувствительные** ключи в `.env.example` заменяются на плейсхолдеры вида `your_..._here`.

### Категоризация ключей

**Чувствительные ключи** (должны быть плейсхолдерами в `.env.example`):

- `BOT_TOKEN`, `API_TOKEN`, `SECRET_KEY`
- `PROXY_URL` (если содержит пароль/адрес)
- `ADMIN_IDS` (персональные ID пользователей)
- `DATABASE_URL` (если содержит пароль)
- Любые ключи доступа к внешним сервисам (Sentry DSN, NTFY URL, etc.)

**Нечувствительные ключи** (реальные значения в `.env.example`):

- `SQLITE_DB_PATH`, `CACHE_PATH`
- `PYTHONUTF8`
- `LOG_LEVEL`, `ENVIRONMENT`
- Таймауты, интервалы, пороговые значения

### При добавлении нового ключа

1. Добавить ключ с реальным значением в `.env`.
2. Добавить ключ в `.env.example`:
   - Если чувствительный — плейсхолдер `your_..._here`.
   - Если публичный — реальное значение.
3. Убедиться, что `pydantic_settings` в `config.py` подхватывает новый ключ.

## Markdown (.md)

### Иерархия заголовков

- **Один H1** (`#`) на файл.
- Строго последовательная иерархия: H1 → H2 → H3 (без пропусков уровней).

### Блоки кода

- Обязательное указание языка: `python`, `json`, `bash`, `text`, `mermaid`, `yaml`.

### Списки

- Единый маркер: `-`.
- Пустая строка перед и после списка.

### Пробелы

- Отсутствие висячих пробелов в конце строк.
- Завершение файла ровно одной пустой строкой (`\n` в конце).

### Валидация

- Файлы должны проходить `markdownlint` и `prettier`.
- Форматирование через `npx prettier --write` (proseWrap: preserve, printWidth: 120).
- Проверка markdownlint: `npx markdownlint "docs/**/*.md" ".roo/**/*.md" "*.md"` (ожидается 0 errors).

## Стилистика текста

- Лаконичность, сухость, технический стиль.
- Исключены: эмоции, избыточные описания, вода.
- Только факты, алгоритмы, спецификации.

## Ссылки на файлы

- В markdown-документах ссылки на файлы оформляются с указанием строки: ``[`filename.ext:line`](relative/path/file.ext:line)``.
- Для конструкций кода указывается и строка: ``[`function_name()`](path/file.py:42)``.
