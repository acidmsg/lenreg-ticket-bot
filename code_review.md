**Дата:** 2026-05-14

**Объём:** ~3500 строк Python + ~600 строк тестов

## 1. Архитектура и структура

### 1.1. Общая оценка

Проект построен по чистой слоёной архитектуре с чётким разделением ответственности. Стек: aiogram 3.x (Telegram Bot API), SQLite (WAL, aiosqlite), Redis (aioredis), httpx (AsyncClient), pydantic-settings. Код организован логично, имена пакетов отражают их зону ответственности.

**Сильные стороны:**

- Строгое следование принципу единственной ответственности (SRP): пакеты `api/`, `database/`, `handlers/`, `services/`, `middleware/`, `utils/` изолированы.
- Документированная архитектура в [`ARCHITECTURE.md`](docs/ARCHITECTURE.md) с графом зависимостей Mermaid, таблицей зон ответственности и физическим деревом директорий.
- Централизованный SSOT (Single Source of Truth) в [`docs/openapi.yaml`](docs/openapi.yaml) — все контракты данных, бизнес-правила и схемы БД зафиксированы.
- Правильно выстроенный middleware-конвейер: `ErrorBoundary` → `UserRateLimit` → `UserDataPreload` → `ActivityLog`.

## 2. Code Review и бизнес-логика

### 2.1. Общая оценка

Код в целом чистый, типизированный (mypy-совместимый), с docstrings на русском языке. Используются Pydantic-модели для десериализации ответов API. Асинхронность выдержана последовательно: `asyncio.Lock` для атомарных операций, `aiofiles` отсутствует (файловый ввод-вывод не используется в рантайме), фоновые задачи через `asyncio.create_task`.

### 2.2. Критические замечания по бизнес-логике

| #   | Проблема                                                                                                                                                                                                                                                                                                                                                                                                                                                                               | Файл:строка                                  | Серьёзность |
| --- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------- | ----------- |
| B1  | **Наивный расчёт возраста в [`is_child()`](src/utils/helpers.py:34).** Формула `(datetime.now() - bday).days // 365` не учитывает високосные годы. Для человека, родившегося 2008-05-15, при проверке 2026-05-13 результат будет `17` (ещё 17), а 2026-05-15 — `18`. Но из-за 4-5 високосных лет за 18-летний период погрешность составляет ±4 дня. Не критично для бизнес-логики «взрослый/ребёнок», но потенциально может дать ложный переход на день раньше/позже.                  | `src/utils/helpers.py:34`                    | LOW         |
| B2  | **Глухое подавление исключений в хендлерах.** В [`common.py`](src/handlers/common.py) обнаружено **6 блоков** `except Exception: pass` на строках 227, 277, 404, 466, 509, и ещё несколько в других обработчиках. Это означает, что при ошибке редактирования сообщения (например, `Message is not modified`) пользователь просто не получит ответа. Хотя `ErrorBoundaryMiddleware` перехватывает `TelegramBadRequest`, голые `except Exception: pass` могут скрывать реальные ошибки. | `src/handlers/common.py:227,277,404,466,509` | MEDIUM      |
| B3  | **Отсутствие валидации `callback_data` в `back_to_clinics`.** В [`back_to_clinics()`](src/handlers/common.py:306) префикс `back_to_clinics_` разбирается через `split("_")[3:]`, но нет проверки минимальной длины. Если пользователь каким-то образом отправит callback*data `back_to_clinics*`без суффикса, упадёт`IndexError`.                                                                                                                                                      | `src/handlers/common.py:312-318`             | MEDIUM      |

### 2.3. Качество кода

**Сильные стороны:**

- Pydantic-модели [`src/api/models.py`](src/api/models.py) с алиасами полей (`validation_alias=AliasChoices`) и валидаторами (`_coerce_str`) — грамотная защита от изменений API.
- Rate limiting через `aiolimiter` с раздельными лимитерами для разных типов задач (мониторинг, discovery, healthcheck, пользовательские запросы) — [`src/api/zdrav_client.py:24-29`](src/api/zdrav_client.py:24).
- Retry-логика с 3 попытками для 5xx и сетевых ошибок — [`src/api/zdrav_client.py:102-115`](src/api/zdrav_client.py:102).
- Собственная система миграций БД с версионированием — [`src/database/migrations.py`](src/database/migrations.py:14).
- Корректная обработка `CancelledError` во всех 4 фоновых циклах для graceful shutdown.

## 3. Производительность и работа с данными

### 3.1. Общая оценка

Проект демонстрирует осознанную работу с производительностью: WAL-режим SQLite, connection-level блокировки, Redis для кэширования результатов мониторинга, in-memory кэш для данных пользователей. Фоновые задачи изолированы и не блокируют обработку пользовательских запросов.

### 3.2. Проблемы производительности

| #   | Проблема                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                     | Файл:строка                                                    | Серьёзность  |
| --- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | -------------------------------------------------------------- | ------------ |
| P1  | **Глубокая копия всего кэша при каждом доступе к `db.data`.** Свойство [`DatabaseManager.data`](src/database/manager.py) (через `_get_user_data_nolock`) возвращает словарь, но в [`monitor_loop()`](src/services/monitor.py:97) используется выражение `users_data = db.data`, которое вызывает итерацию по всем пользователям. Это происходит каждые ~300 секунд. `deepcopy` не используется явно, но словарь пересобирается через `dict()` при каждом обращении к `_data_cache`. При 100+ пользователях это может создавать заметную нагрузку на GC.      | `src/services/monitor.py:97` → `src/database/manager.py:41-52` | MEDIUM       |
| P2  | **Линейный обход всех пользователей в monitor_loop.** [`monitor_loop()`](src/services/monitor.py:110-204) обходит всех пользователей, всех пациентов, всех врачей в тройном вложенном цикле. Между проверками `await asyncio.sleep(random.uniform(1.0, 3.0))` — это джиттер, а не реальный параллелизм. При 100 пользователях × 2 пациента × 3 врача = 600 запросов последовательно. При таймауте API 10 секунд полный цикл может занять > 30 минут. **Решение:** использовать `asyncio.gather` с семафором для параллельных проверок в рамках одного цикла. | `src/services/monitor.py:110-204`                              | **CRITICAL** |
| P3  | **Создание нового `httpx.AsyncClient` при каждом запросе, если клиент не создан.** В [`ZdravClient._get_client()`](src/api/zdrav_client.py:59-69) при первом вызове создаётся клиент, но из-за ленивой инициализации возможна гонка между первыми запросами (корутины могут создать несколько клиентов). Хотя на практике это маловероятно (клиент создаётся в `main()` до запуска фоновых задач), защита через `asyncio.Lock` была бы надёжнее.                                                                                                             | `src/api/zdrav_client.py:59-69`                                | LOW          |

### 3.3. Схема БД

Схема грамотно нормализована: отдельные таблицы для пользователей (`users`), пациентов (`user_patients`), мониторинга (`user_monitoring`), последних сообщений (`user_last_messages`), врачей (`doctors`), клиник (`clinics`), конфигурации (`config`), псевдонимов (`specialty_aliases`). Индексы корректны. WAL-режим включён.

## 4. Безопасность и наблюдаемость

### 4.1. Общая оценка

Проект уделяет серьёзное внимание безопасности: все секреты вынесены в `.env`, `.env.example` содержит плейсхолдеры, настроен `.gitignore`. Наблюваемость на хорошем уровне: Sentry + NTFY для ошибок, Loguru для структурированного логирования, healthcheck с метриками.

### 4.2. Проблемы безопасности

| #   | Проблема                                                                                                                                                                                                                                                                                                                                                                                                  | Серьёзность |
| --- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------- |
| S1  | **CSRF-токен захардкожен в `.env.example`.** Значение `CSRF_TOKEN=NOTPROVIDED` в [`.env.example`](.env.example) — это технический токен, который всегда одинаков. Поскольку он не секретный, это допустимо, но в коде он передаётся как заголовок `X-CSRFToken`. Если API здравницы когда-либо начнёт валидировать CSRF, бот сломается. Рекомендуется вынести в настройки с возможностью переопределения. | LOW         |
| S2  | **Отсутствие проверки `Forwarded`/`X-Real-IP` заголовков.** Не применимо — бот работает через Telegram API, а не прямые HTTP-запросы.                                                                                                                                                                                                                                                                     | N/A         |
| S3  | **Redis без пароля.** В [`docker-compose.yml`](docker-compose.yml:2-10) Redis поднимается без `requirepass`. При биндинге на `127.0.0.1` это приемлемо, но если когда-либо потребуется внешний доступ — станет уязвимостью.                                                                                                                                                                               | LOW         |

### 4.3. Наблюваемость

| #   | Проблема                                                                                                                                                                                                                                                                                            | Серьёзность |
| --- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------- |
| S4  | **Метрики мониторинга не экспортируются.** [`HealthMetrics`](src/services/healthcheck.py:24-90) собирает отличные метрики (количество проверок, ошибок API, уведомлений, uptime), но они доступны только через команду `/status` администратору. Нет экспорта в Prometheus/InfluxDB/другие системы. | MEDIUM      |
| S5  | **Sentry инициализируется, но breadcrumbs не настраиваются.** В [`_init_sentry()`](src/services/error_notifier.py:23-42) создаётся `sentry_sdk.init()` без кастомизации `before_send` или `traces_sample_rate`. Это ограничивает полезность Sentry для отладки.                                     | LOW         |
