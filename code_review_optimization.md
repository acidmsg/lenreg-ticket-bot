## 2. Категории находок

### 2.1 Быстрые победы (Low Risk, 10-50 строк)

#### A. Дублирование `nav_type_map` — 3+ копии

**Где:** `src/handlers/common.py`, строки 571-576, 644-649, 738-743, 898-903

```python
nav_type_map = {
    "adult": "doctor_adult",
    "child": "doctor_child",
    "all": "doctor_dentist",
}
```

Определён **4 раза** в одном файле. Каждый раз это 6 строк идентичного кода.

**Рекомендация:** Вынести в константу уровня модуля:
```python
_CLINIC_NAV_TYPE_MAP = {
    "adult": "doctor_adult",
    "child": "doctor_child",
    "all": "doctor_dentist",
}
```
**Экономия:** ~18 строк.

---

#### B. Дублирование выбора города (select_city vs back_to_clinics)

**Где:** `src/handlers/common.py`, строки 405-452 и 482-532

Оба хендлера содержат идентичный код для:
1. Раскодирования `city_idx` → `selected_city` (8 строк)
2. Построения `city_label` (2 строки)
3. Вызова `get_clinic_selection` с теми же аргументами (10 строк)

Разница только в источнике `city_idx` (из `call.data` vs уже известен).

**Рекомендация:** Вынести общую логику в хелпер:
```python
async def _build_clinic_selection_kb(p_id, city_idx, ...):
    ...
```
**Экономия:** ~40 строк.

---

#### C. Дублирование шаблона сообщения о слотах

**Где:** `src/services/monitor.py` (строки 216-233) и `src/handlers/common.py` (строки 708-711)

Практически идентичный паттерн:
```python
f"{spec_text}🧑‍⚕️ {d_name_display}\n"
f"👤 {p_label}\n{header}\n\n"
+ "\n".join(slot_lines) + link
```

**Рекомендация:** Вынести в `utils/helpers.py` функцию `format_notification_text(...)`.
**Экономия:** ~20 строк (но уменьшит дублирование логики).

---

#### D. `doctor_manager.py` — избыточная прослойка (24 строки)

**Где:** `src/database/doctor_manager.py` (весь файл)

`DoctorManager` — это тонкая обёртка, которая:
- Создаётся только для передачи в `discovery_loop`
- `discovery_loop` (строка 61) использует `doctor_manager._db` напрямую
- `merge_doctors` — прокси к `database.merge_doctors`

**Рекомендация:** Удалить `DoctorManager`, заменить на прямые вызовы `Database` в `discovery_loop`.
**Экономия:** ~24 строки + 1 файл.

### 2.2 Умеренные изменения (Medium Risk, 50-200 строк)

#### E. Дублирование send_notification / send_nav_photo

**Где:**
- `src/services/monitor.py` — `_send_notification` (строки 18-53, 36 строк)
- `src/handlers/common.py` — `_send_nav_photo` (строки 156-241, 86 строк)

Обе функции:
1. Удаляют старое сообщение
2. Отправляют новое (send_photo или send_message)
3. Сохраняют message_id в БД

**Рекомендация:** Объединить в единую утилиту `send_pinned_message` в `utils/helpers.py`. Разница только в типе изображения (nav vs notify) — можно параметризовать.
**Экономия:** ~60 строк.

---

#### F. Дублирование построения текста статуса в toggle_doctor

**Где:** `src/handlers/common.py`, строки 670-731

`toggle_doctor` (при включении мониторинга) содержит:
1. Отправку "загрузочного" сообщения (строки 675-678)
2. `api.check_slots()` (строка 681)
3. Сохранение в кэш (строки 684-686)
4. Форматирование результата (строки 691-711)
5. Отправку финального сообщения с изображением (строки 719-731)

Эта логика дублирует логику `monitor_loop`. Можно вынести в общий хелпер `check_and_notify_slots(...)`.

**Экономия:** ~50 строк.

---

#### G. `_start_background_tasks` — избыточность

**Где:** `src/main.py`, строки 182-220

Функция:
1. Запускает `monitor_loop` (1 задача)
2. Читает `clinic_ids` из БД
3. Для каждой клиники запускает `discovery_loop` (N задач)
4. Запускает `healthcheck_loop` и `cleanup_loop`

Можно упростить: вместо запуска per-клиника discovery_loop, запустить один общий discovery_loop, который сам итерирует clinic_ids. Это сэкономит ~20 строк в main.py и 10 строк в doctor_discovery.py.

**Экономия:** ~30 строк.

---

#### H. main.py — логика автоопределения прокси (76 строк)

**Где:** `src/main.py`, строки 42-118

`_generate_docker_gateways` + `_discover_proxy` + `_probe_host` — 76 строк для сканирования Docker gateway'ей. Функционал специфичный, редко используемый.

**Рекомендация:** Вынести в отдельный модуль `src/utils/proxy_discovery.py`.
**Экономия:** ~76 строк из main.py (не сокращение, а декомпозиция).

### 2.3 Архитектурные вопросы (High Risk, 200+ строк)

#### I. Database + DatabaseManager — можно ли объединить?

**Текущая архитектура:**
- `Database` (658 строк) — низкоуровневый SQLite-движок
- `DatabaseManager` (229 строк) — кэширующая обёртка с `asyncio.Lock`
- `DoctorManager` (24 строки) — тонкая прослойка (уже рекомендовано удалить)

**Проблема:**
- 12 методов в DatabaseManager — чистые прокси к Database (ниже)
- Кэш синхронизируется сложно: `update_user` делает read → modify → write + refresh
- Методы `data` (копия всего кэша) — потенциально дорогие

**Методы-прокси в DatabaseManager:**
| Метод DatabaseManager | Прокси к Database |
|----------------------|-------------------|
| `get_doctors_for_clinic` | `get_clinic_doctors` |
| `merge_doctors` | `merge_doctors` |
| `get_clinic_name` | `get_clinic_name` |
| `get_all_clinic_names` | `get_all_clinic_names` |

**Рекомендация:**
1. Слить DatabaseManager в Database (объединить кэш-логику напрямую)
2. Или оставить как есть — разделение оправдано для тестов (можно мокировать DatabaseManager)
3. Убрать методы-прокси, вызывать `db._db` напрямую где нужно

**Экономия:** ~100 строк (если объединить) или ~30 строк (убрать прокси).

---

#### J. healthcheck.py + monitor.py — можно объединить?

**Текущая архитектура:**
- `monitor.py` (210 строк) — проверка слотов и уведомления
- `healthcheck.py` (207 строк) — проверка доступности API

**Анализ:** Функционал ортогонален:
- `monitor_loop` работает со слотами конкретных врачей
- `healthcheck_loop` работает с API в целом

**Вывод:** Объединять не стоит — разные циклы, разная периодичность, разная ответственность.


## 5. Вывод

Кодовая база в хорошем состоянии. Основные проблемы:
1. **Дублирование констант и мелких блоков** (nav_type_map, выбор города) — легко исправить
2. **DoctorManager как избыточная абстракция** — можно удалить без потерь
3. **Дублирование отправки сообщений** — можно объединить в утилиту
4. **DatabaseManager** — спорный, требует обсуждения

Проект не перегружен — соотношение кода к функционалу адекватное. Оптимизация на 5-10% возможна без риска.
